Displaying Hierarchical Data in a Grid (WPF)

Feb 10, 2016 in .NET, .WPF

I stumbled upon a question on StackOverflow. The essence of the question was binding a hierarchical data to a grid component. I found this question interesting and did a quick search. There were many examples on how to display hierarchical data in trees or nested grids but I did not found an example on putting it in a single grid. So I decided to create mine.

I decided to up the difficulty by not writing any custom code in pages or windows and keep everything in MVVM+XAML way. I started by creating a sample hierarchical data parallel to the question. There are sellers and these sellers have items in their stocks. For simplicity I selected primary keys for both suppliers and items to be their names. Items naturally have different prices in different suppliers and these should be editable. To up the challenge a little further let’s assume not all the suppliers have the same items. So;

Supplier1: Item1,Item2,Item3
Supplier2: Item2,Item3
Supplier3: Item1,Item3,Item4

The question stated that new suppliers to be added on runtime.

The next thing is to keep list of suppliers. We want this list to be a singleton and easily accessible. So I used MVVM Light Toolkit which has a handy Locator class. I decided to create random data to experiment. Of course normally this is not the way to go, you want well defined mocks for unit testing.

Now we have our data ready. Turning back to our previous example what we need to do now is to show this data in a grid.

Supplier1: Item1,Item2,Item3
Supplier2: Item2,Item3
Supplier3: Item1,Item3,Item4

into

Supplier1 Supplier2 Supplier3
Item1 50 48
Item2 10 11
Item3 20 9 15
Item4 40

The next problem is representing the empty slots in the grid. A supplier does not know what other suppliers have in their inventories. One way to solve this problem is, to pass the supplier a list of all possible item names and ask for it to return a list of items where the missing item slots simply left blank. Then we can select the proper data templates in XAML for these items. A textbox to edit the price and an empty cell for the missing item. But blanks are harder to create data triggers on and I preferred to rely on data types. So I created a class ItemViewModelFake: ItemViewModel just to give it a different data type.

In WPF you can’t pass data to a binding property. One way to overcome this problem is to use converters. But since the task here is very simple I relied on the Locater and pulled the data from the view model.

The order of items is handled by the MainViewModel and the supplier class respects this order. Now we can select the proper templates in XAML;

To create a proper grid, we need row and column headers. We have all the real suppliers, but to create the rowheader column where we display the distinct item names, we need a new supplier which has all the items.

The hard work is done. Now we need to display this data in a grid like manner. Since we still do not have a tabular data we can’t use a regular datagrid. But we can mimic one with the help of UniformGrid and ItemsControl. The complete XAML now looks like this;

One last thing to do is to handle the creation of suppliers. Our view uses a custom list of suppliers and items. Therefor we need to call the RaisePropertyChanged() events manually. But since event handling and list creation is done in different view models, we need a method to pass the required messages between them. Luckily MVVM Light offers such a mechanism.
